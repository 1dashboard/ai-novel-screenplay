# AI 编剧助理 — 对话记忆系统需求文档

## 1. 概述

当前 AI 编剧助理（ChatSidebar）每次对话都是无状态的：LLM 只看到当前消息，不记得之前讨论过的内容。用户无法进行"把上一版再改改""还是用第一个方案"这类指代性、延续性对话。

本需求旨在为 AI 编剧助理构建完整的对话记忆系统。

## 2. 核心需求

### 2.1 短期记忆（Short-term Memory）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| STM-01 | 同一对话 session 内，AI 能记住最近的 N 轮（默认 20 轮）对话内容 | P0 |
| STM-02 | 上下文窗口可配置，超出窗口的消息自动裁剪（保留最近 + 摘要） | P1 |
| STM-03 | 对话支持多轮指代：如"刚才那个修改再调整一下" | P0 |
| STM-04 | 用户刷新页面或重新打开后，短期记忆自动恢复 | P1 |

### 2.2 长期记忆（Long-term Memory）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| LTM-01 | 所有对话历史持久化存储，不会因服务重启丢失 | P0 |
| LTM-02 | 历史消息支持分页查询，用户可回顾之前对话 | P1 |
| LTM-03 | 对同一剧本的多次对话 session 之间可关联（如"继续上次的修改"） | P2 |
| LTM-04 | 对话可被导出（JSON/Markdown） | P2 |

### 2.3 隔离性（Isolation）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| ISO-01 | 用户 A 的对话完全不可被用户 B 访问 | P0 |
| ISO-02 | 同一用户的不同任务（task_id）之间的对话完全隔离 | P0 |
| ISO-03 | 每个 task 只有一个对话 session（一对一关系） | P0 |
| ISO-04 | API 层强制校验：当前用户 = session 所属用户 | P0 |

### 2.4 消息类型

| 类型 | 说明 | 存储内容 |
|------|------|----------|
| `user` | 用户输入的自然语言指令 | 原始文本 |
| `assistant` | AI 的回复 | 回复文本 + 可选：修改后的 YAML、变更摘要、变更列表 |
| `system` | 系统事件（仅前端展示，不持久化到上下文） | 如"修改已应用""错误：xxx" |

### 2.5 性能要求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| PERF-01 | 加载最近 20 条对话历史的延迟 < 50ms | P1 |
| PERF-02 | 写入消息延迟 < 100ms | P1 |
| PERF-03 | 支持 100+ 并发用户同时对话 | P2 |
| PERF-04 | Redis 不可用时自动降级到纯 DB 模式 | P1 |

## 3. 用户场景

### 场景 1：多轮迭代修改

```
用户: 把第二幕萧炎的台词改得更愤怒一点
AI:   已将场景4中萧炎的对白语气调整得更加愤怒...[显示差异]
用户: 太愤怒了，稍微收一点
AI:   已调整，保留了愤怒感但去掉了过激的表达...[显示差异]
用户: 还要把萧媚的心理活动展现出来
AI:   已在第二幕新增过场戏，展示萧媚的内心独白...[显示差异]
```

**关键点**：第 3 条指令"还要..."依赖于前两条的上下文——AI 需要知道（1）前面已经改了什么（2）"还要"是指对同一个剧本。

### 场景 2：跨 session 恢复

```
用户: (关闭页面，第二天打开同一任务)
      点击"AI 助理" → 看到昨天的对话历史
用户: 继续，把第三幕的场景2也改一下
AI:   (基于历史上下文理解"也"指什么) 好的，沿用上一轮的风格调整...
```

### 场景 3：多任务隔离

```
用户: 在任务A的AI助理中讨论"萧炎的愤怒"
      切换到任务B，打开AI助理 → 应该是一个全新的对话
      切回任务A → 之前的对话还在
```

## 4. 数据结构需求

### 4.1 对话 Session

- 一个用户 + 一个任务 = 一个对话 Session（一对一）
- Session 包含：session_id、user_id、task_id、创建时间、更新时间
- 可选：标题（从首条用户消息自动提取）

### 4.2 消息记录

- 每条消息关联到一个 session
- 消息角色：user / assistant / system
- 用户消息：纯文本
- AI 消息：文本 + 结构化数据（修改的 YAML、变更摘要、变更列表、接受/拒绝状态）

### 4.3 上下文构建

- 从所有消息中选取最近 N 轮（可配置，默认 20）
- 格式化为 LLM 可理解的对话格式
- 用户消息原文 + AI 回复的变更摘要（避免把整个 YAML 塞入上下文）

## 5. 非功能性需求

| 类别 | 要求 |
|------|------|
| 安全性 | 所有 API 端点验证 JWT + user_id 匹配 |
| 可扩展性 | Redis 作为可选组件，可热插拔 |
| 可观测性 | 记录每次 LLM 调用的 token 消耗、延迟 |
| 数据隐私 | 支持按用户/按任务删除全部对话数据 |
